# 07 应用创建向导逐页说明

## 向导总览

当前项目的应用创建向导分为五步：

1. 应用程序。
2. 选择提供程序。
3. 配置提供程序。
4. 配置绑定。
5. 检查与提交应用程序。

最终提交时，系统会创建 Application、Provider，并写入 Policy/User/Group Bindings。

## 第 1 页：应用程序

这一页决定用户在应用库中看到什么。

### 应用名称

显示在应用库中的名字。

建议：

- 写业务名称。
- 小白用户能看懂。
- 不要只写技术代号。

示例：

- 测试系统。
- 内部 Wiki。
- 项目管理平台。

### URL 标识

也叫 Slug，是 URL 中使用的内部名称。

建议：

- 英文小写。
- 用短横线分隔。
- 不要用中文、空格、特殊符号。

示例：

- `test-system`
- `internal-wiki`
- `project-management`

### 组

这是应用展示分组。

它不会控制权限，只会让相同组名的应用显示在一起。

示例：

- 协作。
- 沟通。
- 内部。
- 运维。
- 财务。

### 策略引擎模式

决定多个访问绑定怎么合并。

| 模式 | 说明 |
| --- | --- |
| ANY | 只要任意绑定通过，就允许访问 |
| ALL | 必须所有绑定都通过，才允许访问 |

新手建议：

- 普通应用：ANY。
- 高敏感应用：ALL。

### UI Settings

| 字段 | 作用 | 建议 |
| --- | --- | --- |
| Launch URL | 用户点击应用后打开的地址 | 不知道时可先留空，让系统从 Provider 推断 |
| Open in new tab | 是否新标签页打开 | 外部业务系统建议开启 |
| Icon | 应用图标 | 用清晰图标帮助用户识别 |
| Publisher | 发布者 | 可写部门或公司 |
| Description | 描述 | 写一句话说明用途 |

## 第 2 页：选择提供程序

这一页选择协议类型。

你要先知道目标应用支持什么协议。

常见选择：

- 支持 OIDC：选 OAuth2/OIDC。
- 支持 SAML：选 SAML。
- 不支持 SSO：选 Proxy。
- 要同步用户：选 SCIM。
- 网络设备：选 RADIUS。

## 第 3 页：配置提供程序

这一页根据 Provider 类型展示不同字段。

关键原则：

- 目标应用给你的参数，要原样填到 authentik。
- authentik 生成的 Client ID、Client Secret、Metadata、证书等，要填回目标应用。
- 两边配置必须互相匹配。

最常见要核对：

| 协议 | 必查字段 |
| --- | --- |
| OAuth2/OIDC | Client ID、Client Secret、Redirect URI、Scopes |
| SAML | ACS URL、Entity ID、证书、NameID、Property Mappings |
| Proxy | External host、Internal host、Cookie domain、Outpost |

## 第 4 页：配置绑定

这一页决定谁可以访问应用。

可以绑定三类对象：

| 绑定对象 | 作用 |
| --- | --- |
| Policy | 满足某个策略才能访问 |
| Group | 某个用户组可以访问 |
| User | 某个用户可以访问 |

绑定字段：

| 字段 | 作用 |
| --- | --- |
| Enabled | 是否启用这个绑定 |
| Negate result | 是否反转结果 |
| Order | 执行顺序 |
| Timeout | 检查超时时间 |
| Failure result | 策略执行失败时算通过还是失败 |

新手建议：

- 优先绑定 Group。
- 少用单个 User 绑定。
- `Negate result` 不懂不要开。
- `Failure result` 一般保持失败更安全。
- `Order` 用默认值即可。
- `Timeout` 用默认值即可。

## 第 5 页：检查与提交应用程序

提交前检查：

1. 应用名是否给用户看得懂。
2. Slug 是否稳定。
3. 展示组是否合适。
4. Provider 类型是否正确。
5. Provider 关键参数是否和目标应用一致。
6. 绑定是否正确。
7. ANY/ALL 是否符合预期。
8. 图标、描述、打开方式是否合适。

提交后验证：

1. 管理员确认应用创建成功。
2. 普通测试用户登录 User 界面。
3. 看应用是否出现。
4. 点击应用测试完整登录。
5. 如果失败，进入 Events 查看日志。

## 小白最佳实践

第一次接应用时，不要一次把安全条件加满。

建议顺序：

1. 先创建 Application 和 Provider。
2. 只绑定一个测试用户组。
3. 用测试用户完成登录。
4. 再逐步增加 MFA、IP、风险策略等限制。
