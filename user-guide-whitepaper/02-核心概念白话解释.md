# 02 核心概念白话解释

## 先记住一条主线

用户看到的是 Application，Application 背后使用 Provider，登录过程由 Flow 和 Stage 组成，是否允许访问由 Binding 和 Policy 判断。

## Application

Application 是“应用入口”。

用户在 User 界面的应用库里看到的一个个卡片，就是 Application。

它负责回答：

- 这个应用叫什么？
- 显示在哪个分组？
- 图标是什么？
- 描述是什么？
- 点击后打开哪里？
- 它关联哪个 Provider？

注意：Application 本身不是目标业务系统，它只是 authentik 里的入口和展示对象。

## Provider

Provider 是“登录协议配置”。

不同业务系统支持不同登录协议，所以 Provider 决定 authentik 用什么方式和业务系统对接。

常见 Provider：

| Provider | 适合场景 |
| --- | --- |
| OAuth2/OIDC | 现代 Web 应用，常见优先选择 |
| SAML | 企业 SaaS、老牌系统 |
| LDAP | 目录服务、旧系统兼容 |
| Proxy | 应用自己不支持 SSO，由 authentik 代理保护 |
| RADIUS | VPN、网络设备、无线认证 |
| SCIM | 用户和组自动同步 |
| RAC | 远程访问 |

## Source

Source 是“外部身份来源”。

如果用户不是直接在 authentik 里创建，而是从外部系统登录或同步，就会用到 Source。

例子：

- LDAP Source：从 LDAP/AD 获取用户。
- OAuth Source：用外部 OAuth 登录。
- SAML Source：用外部 SAML 身份源登录。
- SCIM Source：通过 SCIM 同步用户和组。

## User

User 是用户账号。

它代表一个能登录 authentik 的人或账号。用户可以属于多个 Group，也可以有 MFA 设备、令牌、属性和权限。

## Group

Group 是用户组。

它通常用于批量授权。比如把用户加入“研发部”，再把某个应用授权给“研发部”，这样组内所有用户都能访问。

注意：应用创建页面里的“组”字段不是这里的用户组，它只是应用展示分组。

## Flow

Flow 是流程。

它定义一件事要经过哪些步骤。比如登录流程、授权流程、登出流程、用户设置流程。

## Stage

Stage 是流程里的一个步骤。

例如：

- 输入用户名。
- 输入密码。
- 邮箱验证。
- MFA 验证。
- 同意授权。
- 写入用户信息。
- 拒绝访问。

## Policy

Policy 是策略。

它用于判断是否通过某个条件。比如：

- 用户是否属于某个组。
- IP 是否在公司内网。
- 用户风险分是否过高。
- 当前时间是否允许访问。
- 表达式是否返回通过。

## Binding

Binding 是绑定关系。

它把 User、Group 或 Policy 绑定到某个目标对象上，比如应用。绑定后，authentik 才知道访问这个应用时要检查哪些条件。

## Policy Engine Mode

策略引擎模式决定多个绑定条件怎么合并。

| 模式 | 含义 |
| --- | --- |
| ANY | 任意一个绑定条件通过，就允许访问 |
| ALL | 所有绑定条件都通过，才允许访问 |

新手建议：

- 普通应用先用 ANY。
- 高敏感应用用 ALL。

## Events

Events 是事件和审计日志。

登录失败、授权失败、用户修改、后台任务、通知等信息都可以从这里排查。

新手遇到问题，先看 Events。
