# 09 权限和策略：Policies

## Policy 是什么

Policy 是判断规则。

它负责回答：“这个请求是否应该通过？”

Policy 可以绑定到应用、流程、阶段等对象上，用来决定用户能不能继续。

## Policy Binding 是什么

Binding 是绑定关系。

在应用创建向导中，可以把三类对象绑定到应用：

- Policy
- Group
- User

authentik 检查访问权限时，会运行这些绑定。

## Group Binding

Group Binding 表示某个用户组可以访问。

例子：

- 研发组可以访问 GitLab。
- 财务组可以访问财务系统。
- 管理员组可以访问监控系统。

这是最推荐的新手用法。

## User Binding

User Binding 表示某个具体用户可以访问。

适合临时测试或特殊例外。

不建议大量使用，因为后续很难维护。

## Policy Binding

Policy Binding 表示必须通过某个策略。

策略可以做更复杂的判断，例如：

- IP 地址是否在允许范围内。
- 用户是否满足表达式条件。
- 密码是否过期。
- 风险评分是否过高。
- 国家或地区是否允许。

## ANY 和 ALL

Application 上的策略引擎模式会决定多个绑定怎么合并。

| 模式 | 判断方式 | 例子 |
| --- | --- | --- |
| ANY | 任意一个绑定通过即可 | 用户属于研发组，或者满足 IP 策略 |
| ALL | 所有绑定都必须通过 | 用户属于研发组，并且满足 IP 策略 |

## Negate result

`Negate result` 表示反转绑定结果。

例如：

- 原本通过，反转后变成不通过。
- 原本不通过，反转后变成通过。

新手不建议开启，除非你非常确定自己在做“排除条件”。

## Failure result

`Failure result` 表示策略执行失败时，系统把它当成通过还是失败。

安全角度通常建议失败就算失败。

## Order

Order 是执行顺序。

当有多个绑定时，数字可以控制先后顺序。新手一般保留默认即可。

## Timeout

Timeout 是策略检查超时时间。

如果策略执行太久，超过这个时间就停止。新手一般保留默认即可。

## 推荐配置示例

### 普通业务系统

- 绑定：对应业务用户组。
- 模式：ANY。

### 运维后台

- 绑定：运维组。
- 绑定：MFA 策略。
- 绑定：公司内网策略。
- 模式：ALL。

### 临时测试应用

- 绑定：测试用户组。
- 模式：ANY。
- 测试完成后删除或收紧绑定。

## 常见错误

| 错误 | 后果 | 处理 |
| --- | --- | --- |
| 把展示组当权限组 | 权限没有变化 | 使用 Group Binding |
| ALL 条件过多 | 用户无法访问 | 先拆开测试每个条件 |
| Negate 误开 | 权限结果反了 | 关闭 Negate 再测 |
| 给个人用户大量授权 | 后期难维护 | 改成用户组授权 |
| 策略失败算通过 | 安全风险 | Failure result 保持失败 |
